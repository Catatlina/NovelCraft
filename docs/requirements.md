> 日期：2026-07-06

## 一、产品定义

### 1.1 一句话描述
AI 驱动的全流程小说写作平台——从扫榜选题到百万字长篇生成，具备状态机管控、统一记忆中枢、伏笔追踪、多维质量把关、规模化调度、爆款分析与反馈进化能力，覆盖创作、管理、分析、出海发布全链路，UI 对标番茄作家助手，商用级。

### 1.2 能力融合来源（三个开源项目，不含 AI Workbench）

| 来源 | Star数 | 吸收能力 |
|------|--------|----------|
| AI_NovelGenerator | 5.5k⭐ | 多章节伏笔衔接机制（架构参考，非直接调用代码） |
| AI-automatically-generates-novels | 900⭐ | 批量生成流程 + 拆文仿写逻辑 |
| harnessNovel | 40⭐ | 基于 DeepSeek 的拆书流水线思路 |

> 以上均为架构/思路参考，非直接依赖其代码库运行。7 Prompt 引擎、13平台扫榜均需在本项目内自研实现（见第五节）。

### 1.3 产品形态
| 端 | 形式 | 地址 |
|----|------|------|
| Web | React 18 + TypeScript + Vite + Tailwind（构建为静态 SPA 亦可直接 serve dist/） | `http://localhost:5173` (dev) / `http://localhost:8080` (prod via nginx) |
| API | FastAPI REST | `http://localhost:8100` |
| 数据库 | PostgreSQL 16 + pgvector（新增，供知识库向量检索用） |
| 任务队列 | Celery + Redis | 承载调度流水线各队列 |
| 发布自动化 | n8n + Playwright | 海外平台自动登录发布 |

### 1.4 设计语言
主色 #FF6B35 渐变 #FF2442，背景 #F0F2F5，卡片纯白14px圆角，PingFang SC/Microsoft YaHei（正文 Georgia 衬线），暗色模式跟随系统，移动端底部Tab+PC左侧导航。

## 二、核心架构层（❗必须最先建成，其余能力均依赖于此）

### 2.1 小说状态机（❗①）

防止创作流程混乱，每本书必须处于明确状态之一，且状态迁移必须校验前置条件。

```
Idea → Outline → World → Writing → Review → Publish
```

| 状态 | 允许的操作 | 迁移条件 |
|------|-----------|----------|
| Idea | 灵感输入、扫榜选题 | 生成初步书名+一句话简介后 → Outline |
| Outline | 生成/编辑全书总纲 | 总纲通过基本完整性检查（卷数/字数目标已设定）后 → World |
| World | 编辑术语表/力量体系/世界规则 | 知识库核心字段非空后 → Writing |
| Writing | 生成/续写章节 | 每章写完自动进入 Review，全书写完可整体进入 Publish |
| Review | 审查评分、去AI味、人工确认 | 审查分数达标或人工强制通过 → 可回到 Writing（返修）或进入 Publish |
| Publish | 发布至各平台 | — |

- 非法状态跳转需在 API 层拦截（例如 World 阶段未完成不能触发批量生成）。
- 状态机需持久化在 `novel_projects.status` 并记录状态变更历史（用于排查流程异常）。

### 2.2 Context Hub 统一记忆中枢（❗②）

解决长篇生成中 AI 断片、人物OOC、设定冲突的核心组件。所有生成请求（单章/续写/批量）必须经过 Context Hub 组装上下文，而非直接拼接 prompt。

**组装的 7 层上下文**：
1. 全书总纲摘要
2. 当前卷/阶段大纲
3. 人物状态（性格/关系/当前处境，含动态变化追踪）
4. 世界设定（力量体系/世界规则相关片段，按需检索，非全量塞入）
5. 伏笔池（见2.3，哪些伏笔已埋未收，本章是否需要处理）
6. 前5章摘要（非全文，避免上下文过长）
7. 防崩提醒（本章禁止事项：如某角色已死不能出现、时间线不能倒退等）

**技术实现要点**：
- 世界设定/术语表等大体量知识库内容需支持**向量检索**（pgvector），按当前章节相关性动态截取，而非每次全量注入 prompt，否则百万字级项目上下文会爆炸。
- Context Hub 输出应为结构化 JSON，供 7 Prompt 引擎统一消费，而不是各自拼接字符串。

## 三、差异化核心能力（❗③-❗⑧，构建在核心架构层之上）

### 3.1 伏笔系统（❗③，商业级差异点）

| 功能 | 说明 |
|------|------|
| 伏笔埋点 | 生成章节时，AI 主动标记本章埋下的伏笔（存入伏笔池，含伏笔描述/预期回收章节范围） |
| 伏笔回收检测 | 每章生成后自动比对伏笔池，检测本章是否回收了某个伏笔 |
| Payoff 检测 | 判断伏笔回收是否有效（回收内容与埋点预期是否匹配，是否有"水过"嫌疑） |
| 伏笔看板 | UI 展示全书伏笔状态：未回收/已回收/超期未回收 |

### 3.2 7维质量系统升级（❗④）

| 维度 | 说明 | 处理方式 |
|------|------|----------|
| 一致性 | 人物设定/时间线/世界规则是否自洽 | 检测+标注冲突点 |
| AI味检测 | 识别AI生成的套路化表达 | 检测+提供去AI味建议 |
| 节奏 | 章节内张弛节奏是否合理 | 检测+建议调整点 |
| 人物OOC | 角色言行是否偏离设定性格 | 检测+标注偏离片段 |
| 爽点密度 | 单位字数内爽点分布是否达标 | 检测+密度评分 |
| 对话质量 | 对话是否自然、是否推动剧情 | 检测+标注生硬对话 |
| 结尾钩子 | 章末是否有足够悬念驱动读者翻页 | 检测+钩子强度评分 |
| **自动重写修复** | 对不达标维度，AI 自动定向重写对应片段（而非整章重来） | 精确定位问题片段并局部重写，保留其余内容不变 |

> 与旧版差异：旧版只有"审查+建议展示"，v7.0 要求**审查结果可直接驱动定向自动重写**，形成闭环。

### 3.3 自动调度流水线（❗⑤，规模化核心）

五级任务队列，构建在 Celery 基础上：

| 队列 | 职责 |
|------|------|
| Idea队列 | 批量处理灵感输入/扫榜选题，产出候选书名+简介 |
| 大纲队列 | 批量生成/校验全书总纲 |
| 章节队列 | 批量生成正文（含 Context Hub 组装） |
| 审核队列 | 批量跑 7维质量检测+自动重写 |
| 发布队列 | 批量执行翻译+平台发布 |

- 每个队列独立监控、独立重试策略，互不阻塞。
- 队列间通过状态机节点衔接（如某本书从 Writing 完成进入 Review，自动进入审核队列）。

### 3.4 爆款分析系统（❗⑥，赚钱关键）

| 功能 | 说明 |
|------|------|
| 标题结构拆解 | 分析爆款标题的句式结构、关键词组合规律 |
| 开头Hook分析 | 拆解爆款开篇前500字的抓人手法 |
| 爽点密度分析 | 对标爆款章节的爽点分布密度，作为3.2节评分基准 |
| 市场热度评分 | 结合扫榜数据给出题材/书名的市场热度预测分 |

> 该系统的输出（如爽点密度基准）应反馈进 3.2 节质量系统的评分标准，而不是孤立展示。

### 3.5 反馈学习闭环（❗⑦，进化能力）

```
阅读数据 → 分析 → 优化Prompt → 再生成
```

| 功能 | 说明 |
|------|------|
| 阅读数据接入 | 从已发布平台回抓阅读量/追读率/评论等数据 |
| 数据分析 | 关联具体章节内容特征（伏笔密度/爽点密度/节奏）与阅读表现 |
| Prompt优化建议 | 基于分析结果生成 Prompt 调整建议（人工确认后生效，避免自动改坏） |
| 再生成验证 | 优化后的 Prompt 用于后续章节生成，形成迭代 |

> 此模块依赖出海发布/平台数据回流已经跑通，建设顺序上应在其之后。

### 3.6 知识库升级：推理系统（❗⑧）

| 能力 | 说明 |
|------|------|
| 可检索（向量） | 术语表/世界规则等存入 pgvector，支持语义检索，供 Context Hub 按需调用 |
| 可推理（规则） | 支持定义规则（如"力量体系等级不可逆增长"），生成时自动校验是否违反 |
| 可继承（跨小说复用） | 同一作者的多本书可共享/继承部分世界观设定，减少重复搭建 |

## 四、基础创作流水线（在核心架构层之上运行的具体功能）

| # | 功能 | 说明 |
|---|------|------|
| 4-1 | 扫榜分析（13平台，自研） | 产出选题推荐，接入爆款分析系统(3.4) |
| 4-2 | 自动书名生成 | 选平台+题材→AI自动书名 |
| 4-3 | 全书总纲（百万字级，5-8卷） | 由 Context Hub 提供基础上下文 |
| 4-4 | 黄金三章细纲 | 极其详细，供后续章节生成对齐 |
| 4-5 | 第一章正文（2000-3500字） | — |
| 4-6 | 灵感生成 | 一句话 idea → 书名+大纲+细纲+第1章 |

## 五、7 Prompt 引擎（自研，脱离 AI Workbench 依赖）

| Prompt | 名称 | 功能 | 归属 |
|--------|------|------|------|
| novel-scan | 扫榜分析 | 13平台趋势+选题推荐 | 自研，需重新设计信源接入方式 |
| novel-analyze | 拆文学习 | 黄金三章+人设+节奏+文风拆解 | 自研 |
| novel-write | 长篇写作 | 原创/仿写/续写三模式 | 自研，需与 Context Hub 输出格式对齐 |
| novel-short-write | 短篇写作 | 5000-20000字短篇 | 自研 |
| novel-deslop | 去AI味 | 洗稿/润色/去AI三模式 | 自研，输出需可驱动3.2节定向重写 |
| novel-translate | 翻译出海 | 6平台格式适配 | 自研 |
| novel-review | 多视角审查 | 对应3.2节7维度 | 自研，需从原4类扩展至7维 |

> ⚠️ 重要提醒：原扫榜"45信源"能力挂在 AI Workbench 下，现该模块移除后，13平台扫榜的具体数据获取方式（爬虫/API/人工整理）需要单独确定，否则 4-1 无法真正跑起来，这是当前最大的未知项，建议在 Phase 0 阶段就明确技术方案。

## 六、建设路线图（分 Phase，核心架构优先）

| Phase | 目标 | 关键内容 |
|-------|------|----------|
| Phase 0 | 基础设施 + 数据源方案确定 | 建表、FastAPI/前端骨架、DeepSeek联通、13平台扫榜数据源技术方案敲定 |
| Phase 1 | 核心架构层 | 状态机(2.1) + Context Hub基础版(2.2，含pgvector) |
| Phase 2 | 最小写作闭环 | 4-1~4-7，验证 Context Hub 是否真正解决断片问题 |
| Phase 3 | 质量系统 | 3.2 七维质量系统+自动重写；3.1 伏笔系统 |
| Phase 4 | 调度流水线 | 3.3 五级队列 + 4-8批量生成 + 成本控制 |
| Phase 5 | 知识库升级 | 3.6 向量检索+规则推理+跨小说继承 |
| Phase 6 | 出海发布 | 翻译流水线/账号托管/Playwright发布 |
| Phase 7 | 爆款分析 | 3.4，反哺质量系统评分基准 |
| Phase 8 | 反馈学习闭环 | 3.5，依赖 Phase 6 发布数据回流 |
| Phase 9 | 运营与长尾 | 分析看板/版本管理/搜索筛选/导出格式 |

**关键排序逻辑**：状态机和 Context Hub（Phase 1）必须先于一切生成类功能，否则伏笔系统/质量系统/调度流水线都没有稳定的上下文基础和流程管控依托，会导致后期返工。

## 七、技术架构

```
┌───────────────────────┐
│   index.html (SPA)      │  前端
├───────────────────────┤
│   FastAPI :8100         │  后端 + 状态机校验层
├───────────────────────┤
│   Context Hub 服务       │  独立服务/模块：7层上下文组装（新增，核心）
├───────────────────────┤
│   Celery + Redis         │  五级调度队列（Idea/大纲/章节/审核/发布）
├───────────────────────┤
│   PostgreSQL + pgvector  │  数据 + 知识库向量检索（新增 pgvector）
├───────────────────────┤
│   DeepSeek API           │  文本生成
│   Playwright + n8n       │  海外平台自动发布
└───────────────────────┘
```

### 数据库新增/调整
- `novel_projects` 新增：`state_history`（JSONB，状态机迁移记录）
- `foreshadow_pool` 独立化为表：id, project_id, description, planted_chapter, expected_payoff_range, status（未回收/已回收/超期）
- `knowledge_embeddings`（新表）：knowledge_type, project_id, content, embedding（vector类型，pgvector）
- `quality_reviews`（新表）：chapter_id, dimension（7维之一）, score, issues_json, rewrite_applied
- `feedback_signals`（新表，Phase 8用）：chapter_id, platform, read_count, retention_rate, collected_at

## 八、当前完成度

全部功能（含核心架构层）均为未实现状态，本版不再以"完成率百分比"衡量——核心架构（状态机+Context Hub）一旦缺失，后续功能即使"做完"也可能因为没有统一上下文管控而质量不可控。建议以**第六节路线图的 Phase 是否验收通过**作为唯一进度衡量标准。
